# 서강대 자기소개서 자동 작성 서비스 — 기획/설계 문서 (Draft)

> 상태: v0.2 — 9절 질문에 대한 답변을 반영해 MVP(1~3단계)를 구현했습니다. 실제 코드는 이 문서와 같은 `app/` 디렉토리에 있습니다.

## 1. 한 줄 정의

서강대 재학생/졸업생이 이력서를 업로드하면 학사 정보를 자동으로 채워주고, 지원할 공고 링크를 입력하면 해당 공고의 모집요강을 분석해 자기소개서(자소서)·지원동기·강점 항목을 초안으로 생성하며, 각 문단 옆에 "왜 이렇게 썼는지" 근거를 코멘트로 보여주고 항목별로 재생성할 수 있는 웹 서비스.

## 2. 대상 사용자

- 서강대학교 재학생 및 졸업생 (자소서를 처음 써보는 저학년부터, 여러 공고에 지원하는 취준생까지)
- 1인당 여러 개 이력서 버전 / 여러 공고 지원을 관리할 수 있어야 함

## 3. 핵심 사용자 흐름 (Happy Path)

1. **회원가입/로그인** (⚠️ 서강대 이메일(@sogang.ac.kr) 인증으로 재학 여부를 확인할지, 일반 이메일 가입도 허용할지 확인 필요)
2. **이력서 업로드** (PDF/DOCX)
   → 서버가 텍스트를 추출하고 LLM으로 구조화 파싱 (학교, 학과, 학번, 학점, 졸업/재학 상태, 경력, 활동, 어학점수, 자격증 등)
   → 파싱 결과를 폼에 자동 채움 → 사용자가 검수/수정 후 저장
3. **지원 공고 등록**
   → 공고 URL 입력 → 서버가 페이지를 가져와 본문 텍스트 추출 → LLM으로 "모집요강 분석" (직무, 자격요건, 우대사항, 인재상/핵심가치, 자소서 문항 목록 등 구조화)
   → ⚠️ 사람인/잡코리아/원티드/기업 자체 채용페이지 등 사이트별로 크롤링 난이도·robots.txt·로그인 필요 여부가 다름 → 초기 버전은 "공고 본문을 텍스트로 붙여넣기"를 백업 입력 수단으로 반드시 제공 권장
4. **자소서 초안 생성**
   → 이력서 구조화 데이터 + 공고 분석 데이터를 프롬프트에 결합해 문항별(자소서 항목이 있으면 그 항목대로, 없으면 지원동기/성장과정/직무역량·강점/입사 후 포부 등 표준 항목) 초안 생성
   → 각 문단 옆(사이드 패널)에 "이 문단을 이렇게 쓴 이유" 코멘트 표시
     - 예: "공고에서 '데이터 기반 의사결정'을 우대사항으로 명시 → 이력서의 OO 프로젝트(데이터 분석 동아리) 경험을 강조"
5. **항목별 재생성**
   → 사용자가 특정 문항/문단만 선택해 "다시 생성" (톤 조정, 강조 포인트 변경, 글자수 제한 등 옵션 지정 가능)
   → 재생성 시에도 이유 코멘트 갱신
6. **내보내기**
   → 최종본을 텍스트/Word로 다운로드 (⚠️ 각 기업 자소서 폼에 그대로 붙여넣는 용도이므로 복사하기 버튼이 핵심 UX)

## 4. 화면 구성 (초안)

| 화면 | 설명 |
|---|---|
| 대시보드 | 내 이력서 목록, 진행 중인 지원(공고) 목록 |
| 이력서 업로드/편집 | 업로드 → 파싱 결과 검수 폼 |
| 공고 등록 | URL 입력 또는 본문 붙여넣기 → 분석 결과 요약 카드 |
| 자소서 작성 화면 (핵심) | 좌: 문항별 생성된 자소서 텍스트 (편집 가능) / 우: 문단별 근거 코멘트 사이드패널 / 각 항목 상단에 "재생성" 버튼 + 옵션 |
| 이력서 대비 공고 매칭 요약 | 자소서 화면 상단에 "내 강점 vs 공고 요구사항" 매칭표 (선택적으로 눈에 띄게) |

## 5. 데이터 모델 (초안)

```
User
 - id, email, name, auth_provider

Resume
 - id, user_id, file_url, parsed_json (학교/학과/학번/학점/경력/활동/자격증/어학 등), created_at

JobPosting
 - id, user_id, source_url (nullable), raw_text, parsed_json (직무/자격요건/우대사항/인재상/자소서문항 등), created_at

Application (Resume + JobPosting 매칭 단위)
 - id, user_id, resume_id, job_posting_id, status

CoverLetterSection
 - id, application_id, question_text, content, reasoning_comment, version, char_limit, created_at
 - (재생성 시 version 증가, 이전 버전 히스토리 보관 여부 ⚠️ 확인 필요)
```

## 6. AI 파이프라인 설계

1. **이력서 파싱**: PDF/DOCX → 텍스트 추출(OCR 필요 여부는 스캔본일 때만) → LLM structured extraction (JSON schema로 강제)
2. **공고 분석**: raw_text → LLM structured extraction (직무, 필수/우대 자격요건, 인재상, 자소서 문항 목록, 마감일 등)
3. **자소서 생성**: (이력서 JSON + 공고 JSON + 문항) → LLM 생성, 응답을 `{content, reasoning}` 형태로 구조화 출력하도록 프롬프트 설계 → 코멘트는 생성과 별도 호출이 아니라 한 번의 호출에서 함께 받는 게 비용/지연 면에서 유리
4. **재생성**: 기존 content를 컨텍스트에 포함하고 "다른 각도로/더 간결하게/특정 경험 강조" 등 사용자 지시를 반영해 재생성

## 7. 기술 스택 제안

- Frontend: Next.js (React) + TypeScript
- Backend: Next.js API routes 또는 별도 Node/Python 백엔드
- 파일 파싱: PDF는 `pdf-parse`/`pdfplumber`, DOCX는 `mammoth`
- LLM: Claude API (구조화 출력은 tool use/JSON mode 활용)
- 웹페이지 수집: 서버 사이드 fetch + readability 파싱, 로그인 필요 사이트는 지원 제외하고 안내 문구 표시
- DB: PostgreSQL
- 파일 저장: S3 호환 스토리지

## 8. 비기능 요구사항 / 리스크

- **개인정보 보호**: 이력서에는 민감정보(주민번호 일부, 연락처, 사진 등)가 포함될 수 있음 → 저장 시 암호화, 접근 로그, 삭제 기능 필수
- **저작권/약관 리스크**: 채용 사이트 크롤링이 이용약관 위반 소지가 있을 수 있음 → URL 자동 수집은 "베스트 에포트"로 하고, 실패 시 사용자가 직접 텍스트 붙여넣게 하는 fallback 필수
- **표절/허위 경력 생성 리스크**: LLM이 이력서에 없는 경험을 지어내지 않도록 프롬프트에서 "이력서에 명시된 사실만 사용, 추측 금지" 강제 + 생성 결과에 "이 문장은 이력서의 어느 항목에서 근거했는지" 코멘트로 검증 가능하게 함 (요청하신 코멘트 기능이 이 리스크의 완화 장치 역할도 함)
- **서강대 재학 인증**: 학교 이메일 인증 정도로 충분한지, 별도 학적 API 연동이 필요한지 확인 필요

## 9. 확인된 결정 (Decisions)

1. **학사 정보**: 실제 서강대 학사시스템 연동 없음. `data/mock-students.json`에 가상 학생 10명을 두고, 이력서에서 학번/이름을 추출해 그 10명 안에서만 매칭한다(자동 매칭 실패 시 화면에서 직접 선택).
2. **공고 사이트**: 사람인·잡코리아만 1차 지원. 그 외/수집 실패 시 본문 붙여넣기로 대체.
3. **자소서 항목**: 공고 문항을 따라가지 않고 항상 표준 4항목(지원동기/성장과정/강점(직무역량)/입사 후 포부)을 생성.
4. **서강대 브랜딩**: 로그인 제한 없이 마케팅 포지셔닝으로만 접근. 상단 네비게이션에 서강대 배지(placeholder, 추후 공식 로고로 교체 필요)만 노출.
5. **MVP 순서**: 1(이력서)→2(공고)→3(생성) 순차 구현.

## 10. 구현 현황

위 결정을 반영해 Next.js 앱을 `app/` 디렉토리에 구현 완료:

- 이력서 업로드 → 가상 DB 매칭 → 학사정보 자동 채움 (`app/resume`, `app/api/resume/parse`)
- 공고 URL/본문 분석 → 사람인/잡코리아 전용 수집 + LLM 또는 키워드 기반 폴백 분석 (`app/job`, `app/api/job/analyze`)
- 표준 4항목 자소서 생성 + 근거 코멘트 + 항목별 재생성 (`app/generate`, `app/api/coverletter/*`)
- `ANTHROPIC_API_KEY` 유무에 따라 LLM/휴리스틱 자동 전환 (자세한 내용은 `app/README.md` 참고)

향후 과제로 남겨둔 것: 공고 사이트 셀렉터 유지보수, 서버 DB로의 상태 영속화(현재는 localStorage), 실제 서강대 인증 연동 여부 재검토.
